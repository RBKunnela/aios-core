## 3. Mike Cannon-Brookes
### O Unreasonable Builder

**Cargo:** CEO & Co-Founder, Atlassian
**Expertise:** Product Strategy, Scaling, Enterprise Software

#### Perfil Mental

| Dimensão | Característica |
|----------|----------------|
| **Filosofia** | "Constructive Unreasonableness" |
| **Origem** | Bootstrapped com cartão de crédito |
| **Foco** | Unleashing potential of teams |

#### Mental Models

1. **Constructive Unreasonableness**
   - Desafiar status quo
   - Redesenhar o que não funciona
   - Empoderar outros a fazer o mesmo
   - Baseado em: "All progress depends on the unreasonable man" (Shaw)

2. **Triad Model**
   - PM + Designer + Engineering Lead
   - Se designer e engineer respondem perguntas um do outro = time saudável
   - Se só PM responde = red flag

3. **Deep Knowledge Testing**
   - Líder deve ir "6 níveis deep" em qualquer tópico estratégico
   - Testar profundidade de conhecimento em áreas específicas
   - Não ser pedante, mas verificar domínio real

4. **Subway vs Control Tower**
   - Não tente ser o "control tower" (controlar tudo)
   - Seja um "great citizen" que integra bem
   - Mundo muda rápido demais para controle centralizado

#### Perguntas que Faria

- "Você consegue ir 6 níveis deep nesse assunto?"
- "O time está funcionando como triad ou é só o PM respondendo?"
- "Isso é razoável demais? Onde está a ousadia?"

#### Aplicação para Agentes

**@pm, @po, @architect:**
- Questionar premissas "razoáveis"
- Avaliar saúde do time pela dinâmica, não só outputs
- Ser "bom cidadão" em ecossistemas maiores

2. **The Four Ls (Quarterly)**
   - Learning contínuo
   - Unlearning ativo - livrar-se de conhecimento obsoleto
   - Evolução constante
   - "How do you rid yourself of historical knowledge that won't pay dividend?"

3. **Effectiveness Over Productivity**

4. **Equal Distribution**
   - "When one person is remote, everyone is remote"
   - Nivelar playing field
   - Liberdade permite que pessoas tragam seu melhor

5. **Team Health Checks**
   - "Most valuable tool I've ever used"
   - Times que têm conversas honestas sobre dificuldades = sucesso
   - Auto-awareness + ownership de melhoria

## 5. Maaret Pyhäjärvi
### A Exploratory Testing Master

**Cargo:** Principal Test Engineer, Testing Advocate
**Expertise:** Exploratory Testing, Contemporary Testing, Quality

#### Perfil Mental

| Dimensão | Característica |
|----------|----------------|
| **Filosofia** | All testing is exploratory |
| **Estilo** | Empirical technologist |
| **Diferencial** | Programmer + Tester + Teacher |

#### Mental Models

1. **Testing as Learning**
   - "Whoever tests needs to be learning"
   - Learning deve mudar o que você faz
   - Não separar design de testes da execução

2. **Value Optimization**
   - Value of testing = value of information
   - Value exists in relation to cost (direct + opportunity)
   - Seek to optimize, not maximize

3. **Contemporary Exploratory Testing**
   - Programmatic tests são parte de exploratory testing
   - Combine new tools com traditional principles
   - Test cases são OUTPUT, não input

4. **Continuous Learning Loop**
   - Learn about: application, yourself as tool, other tools, world the app lives in
   - Each test informs the next
   - Reflection is part of the process

5. **Exploratory ≠ Manual**
   - "It isn't manual, it's attended"
   - Automation can call you to attend
   - Not about error guessing - it's systematic

#### Perguntas que Faria

- "O que você aprendeu com o último teste que mudou o próximo?"
- "Seus test cases são input ou output do processo?"
- "Qual é o custo de oportunidade do que você não está testando?"
- "Você está testando ou apenas executando scripts?"

#### Aplicação para Agentes

**@qa, @xray (test management):**
- Tratar testes como processo de aprendizado
- Priorizar por valor de informação
- Documentar learning, não só pass/fail

## 7. Sérgio Freire
### O Test Management Architect

**Cargo:** Head of Solution Architecture & Testing Advocacy, Xray
**Expertise:** Xray, Jira, Test Management, Quality Engineering

#### Perfil Mental

| Dimensão | Característica |
|----------|----------------|
| **Background** | Developer → Tester → Advocate |
| **Filosofia** | Dev and Test as unified team |
| **Foco** | Practical test management that works |

#### Mental Models

1. **Unified Dev + Test**
   - Development and testing should "merge"
   - Act as unique team with common goal
   - Best product stakeholders need

2. **Testing as Understanding**
   - "Testing is a process to understand something"
   - Obtain information to take actions
   - Not limited to time, space, team member, type, or technique

3. **Purpose-Driven Tests**
   - "Reason for a Test to exist should be its purpose/goal"
   - Tests should have necessary steps to validate purpose
   - No more, no less

4. **Quality as Multidimensional**
   - Speed of change increasing
   - Time is one of quality dimensions
   - Handle quality matters wisely

5. **Automation ≠ Goal**
   - "Some think automation is the goal. But is it?"
   - Automation is a means, not end
   - Value of test > how it's executed

6. **Avoid Test Dependencies**
   - Hard to manage tests that depend on previous tests
   - Isolate through Pre-Conditions if needed
   - Independent tests = maintainable tests

#### Perguntas que Faria

- "Qual é o propósito desse teste? Ele precisa existir?"
- "Dev e QA estão trabalhando como um time ou em silos?"
- "Automação é seu objetivo ou sua ferramenta?"
- "Seus testes são independentes ou têm dependências escondidas?"

#### Aplicação para Agentes

**@xray, @qa, @jira:**
- Definir propósito claro para cada teste
- Minimizar dependências entre testes
- Integrar test management com development flow
